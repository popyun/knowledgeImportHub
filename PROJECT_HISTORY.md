> 本文档集中记录 OCR→Markdown 版面还原项目在迭代中 **尝试过但最终回滚、搁置或默认关闭** 的技术方向，作为工程历史留存，避免后续重复踩坑。与之互补的是：`TASK_STATUS.md`（逐迭代流水账与当前状态）、`PROJECT_SUMMARY.md`（当前架构）。

## 总贯约束（贯穿所有尝试）

以下硬约束是多个方案被判定“失败/不采纳”的根本原因，理解历史必须先理解它们：

1. **每张图必须变好或不变差**（零质量下降）；任何改动必须通过全量 33 张回归，默认关闭时与 HEAD 逐字节等价。
2. **免费/离线优先**，不引入任何付费/联网模型；只用已装本地模型。
3. **本机硬件**：i5-11300H、CPU-only（无独显，Intel Iris Xe 闲置）、16GB 内存但空闲常仅 ~4-6GB。这直接决定了本地 VLM 只能跑 1.5B~3B 且极慢。

## 1. 方案 A：PP-Structure 区域增强（迭代十六 / 7.15）

**尝试方向**：对方案 B 上报的低置信表格区域裁剪，调 PaddleOCR PP-Structure（`table=True`）重新识别出结构化表格。

**为何未成主线**：在本套彩色幻灯片语料上实测，PP-Structure 整页 layout 把幻灯片误判成一个 `figure`，切不出表；对低分“伪表格”（121125 相关系数矩阵、121238 宽扁说明块）裁剪后结果比方案 B 更差；对真正并排合并表（121134）裁剪后仍错乱。

**处置**：保留代码但采用“纯附加、绝不替换主输出”形态，增强结果仅作为 `> [!tip] 供人工比对` 附块；默认关闭（`enhance_on_low_quality: false`）。

**教训**：若要让方案 A 真正“增强”而非“比对”，需更强的表格模型，或对彩色表做去底色/网格线预处理（即后来的 gridboost）。

## 2. 方案 A 三档降级 + gridboost 预处理（迭代十七 / 7.16）

**尝试方向**：把单一增强器重构为“可插拔后端 + 三档能力自动降级”。首启探测主机能力（CPU/内存/GPU/库）写 `host_profile.local.json`，据此选档：高配走 vision、中配走 gridboost（纯 CPU 预处理）、低配走 manual。gridboost：去底色二值化 + 用 OCR 词框聚类补虚拟网格线，把“无线表”变“有线表”。

**为何未成主线**：gridboost 预处理后的 PP-Structure 在本语料仍不能稳定优于方案 B（121134 并排合并表 `S_e=0.845 < baseline 0.886`；121125 矩阵、121238 宽块更差）。中配本机探测结论恒为 `gridboost`（空闲内存 < `VISION_MIN_FREE_GB=8.0`、无 GPU）。

**处置**：三档框架、host_profiler、可插拔后端 **保留为主线基础设施**（后续 vision 档与方向 3 都建在其上）；但 gridboost 作为增强手段本身默认关闭、review-only。

## 3. 方案 A vision 档：本地 qwen2.5vl:3b 单区域增强（迭代十八 / 7.17）

**尝试方向**：`ollama pull qwen2.5vl:3b`（本地免费视觉模型），对方案 B 上报的低置信表格区域裁剪，走 `/api/generate` 传 base64 PNG，让 VLM 输出单个 HTML `<table>`。

**部分有效**：`121125`（18×18 相关系数矩阵）vision 档命中低置信区，`S_b=0.431 → S_e=0.857`，比对块结构明显更整齐。

**为何未成主线**：（1）vision 档只对“低置信表格区”触发，`121053`（流程图式混排）/`121134`（并排合并表质量分 0.85 高于阈值）**不触发**；（2）本机 CPU 跑 3B VLM 极慢，单区域 40s~500s+，曾 300s 超时（由 vision_timeout 兜底降级）。

**处置**：VisionLocalBackend 保留为主线可选后端；默认关闭、review-only；本机探测仍为 gridboost，需显式 `backend="vision"` 才走。

## 4. 整页 VLM 重建 + 覆盖率护栏（迭代十九 / 7.18）——已回滚

**尝试方向**：新增 `processors/vision_page.py`（`VisionPageRebuilder`），把复杂混排/低置信整区交给本地 VLM 整页转 Markdown；用覆盖率护栏 `coverage = VLM字符量 / OCR字符量`，`>= 0.75` 才采纳替换正文。

**探针实测**：`121053`（流程图混排）coverage=1.07，结构显著变好；`121134`（宽表）VLM 只吐左上一小块 coverage≈0，被护栏正确拒绝。

**致命反例**：`121131`（正常文本+18 行分组表）coverage=0.79 **通过护栏被采纳，但内容变差**：VLM 把已正确的 18 行压成 15 行、行号错乱、评级与行业列被合并、丢内容，还多出 LaTeX 噪音。

**回滚理由**：覆盖率护栏只能保证“字数够”，拦不住 VLM 在字数达标时**重排/合并/篡改**表格。这违反硬约束“每张图必须变好或不变差”。对 121053 有效仅因其原本结构全丢、怎么改都是改善，不能推广。

**处置**：`git checkout` 恢复 4 个被改文件、删除 `vision_page.py`；代码无残留。

## 5. 方向 3：分区域 VLM 重建 + 逐 token 一致性判据（迭代二十 / 7.20）——已落地、本机未能采纳

## 6.5 OCR 模块升级无损调研（2026-07-07）：PaddleOCR 3.7 三候选

**背景**：评估在现有框架内换 OCR 模块能否提升识别率。现用 `paddleocr==2.7.3 + paddlepaddle==2.6.2`（CPU-only）。先构建隔离环境 `D:\ocr_probe`（venv），**不动主环境**，失败只需删目录即可回退。主环境包快照存于 `_mainenv_baseline.txt`。

**关键发现**：paddleocr 3.7.0 已以 PaddleX 3.x 为底座，PP-OCRv5 / PP-StructureV3 / PaddleOCR-VL 三个候选**共用同一环境**（`paddlepaddle 3.2.0 + paddleocr 3.7.0 + paddlex[ocr] 3.7.2`）。`paddlepaddle 3.0.0` 太旧（PP-OCRv5 报 PIR strides 算子 bug、PaddleOCR-VL 缺 `fused_rms_norm_ext`），升到 3.2.0 后 VL 可加载。

### 候选1：PP-StructureV3（表格结构引擎升级）

- **耗时**（本机 CPU）：首次 init + 模型下载 241s（一次性，缓存后 ~145s）；单张 predict **260s（常规表 121121）~ 585s（大矩阵 121125）**。
- **质量（关键）**：`121121`（常规表格页）输出 **质量优秀**——标题正确、正文分段清晰、两个表格（金融工具 P&L 表 4 列、JTD 示例表）结构与内容均准确。这与旧版 PP-Structure（整页误判为 figure、切不出表，见 7.15）是质的差别。
- **局限**：`121125`（18×18 相关系数矩阵）输出 `S_e=0.944`（行列规整），但 **内容仍错乱堆叠**（数字张冠李戴）——S_e 只测结构规整度、不测内容正确性（同 7.18 教训）。对超密集矩阵，bbox 精度极限仍在。
- **初步结论**：**最值得推进的候选**。对真正的表格页质量明显优于旧版；但单张分钟级耗时，不能入主链，适合做新的可选表格后端（复用 table_enhancer 比对档，review-only）。

### 候选2：PaddleOCR-VL（百度端到端多模态 OCR，0.9B）

- **耗时**（本机 CPU）：权重下载 ~1.8GB；init + 加载 195~249s；单张 predict **>11min 仍未返回**（121134 并排宽表与 121121 常规表均如此，各自独占 CPU 也一样）。
- **结论**：模型在 `paddlepaddle 3.2.0` 上能正常加载（路径打通），**但本机 CPU 推理速度不具备实用性**。这与方向 3（qwen2.5vl:3b）碰到的是同一堵墙：模型可能更对口（专为 OCR 训练），但本机硬件压不动。实际输出质量未能采集（推理未完成），留待 Mac 等更强机器验证。

### 候选3：PP-OCRv5（文字层升级）

- **耗时/可行性**：在 `paddlepaddle 3.0.0` 上报 PIR `strides` 算子类型 bug，`FLAGS_enable_pir_api=0` 未能规避（升 3.2.0 后未再验证，因非重点）。
- **为何优先级最低**：实测现有 2.7.3 的文字置信度已达 **平均 0.94~0.96**（每张仅个位数块 <0.85），文字层已近饱和。难点图的问题都在**版面/表格结构**，不在“字认错”。换 PP-OCRv5 预期整体观感提升很小，不值得为它承担大版本升级风险。

### 综合结论与下一步

1. **PP-StructureV3 是唯一值得推进的**：对真正的表格页（如 121121）质量明显优于旧版，且能复用现有 `table_enhancer` 可插拔后端 + 比对档机制，先 review-only 验证、默认关闭、零回归。代价：需升 paddle 到 3.x（跨大版本，需单独隔离环境或子进程调用）、单张分钟级。
2. **PaddleOCR-VL 潜力大但受 CPU 限制**：同方向 3 的堵墙，适合放到 Mac mini 验证。
3. **PP-OCRv5 优先级最低**：文字层已饱和。
4. **回退方式**：删除 `D:\ocr_probe`（venv）+ `%USERPROFILE%\.paddlex`（模型缓存 ~3.7GB）即可彻底回退；主环境 `paddleocr 2.7.3 / paddlepaddle 2.6.2` 全程未动。本次调研**未修改任何项目代码**。

## 6.6 PP-StructureV3 接入增强后端 + 采纳闸门校准（迭代二十一 / 7.21）——已落地、默认关、本机不触发采纳

**尝试方向**：把 6.5 调研结论中唯一值得推进的 PP-StructureV3 接成现有可插拔增强框架的新后端，作为低质量表格区域的 review-only 增强档；并在此基础上实现“采纳闸门”（当 V3 结果确实明显优于方案 B 时，用 V3 表替换正文表），阈值写入配置文件可调。

**方法**：
- **环境隔离**：主环境保持 `paddleocr 2.7.3`（文字 OCR 已验证逐字节不变）；V3 需 paddleocr 3.x，故通过隔离 venv（`D:\ocr_probe\venv_ppv3`，3.7.0）子进程调用，避开主环境升级带来的 PP-OCRv4→v5 文字层回归风险。新增 `processors/_ppstructurev3_worker.py`（隔离进程入口）。
- **适用性预判**：调用前用几何线索（行/列聚类、每行列数稳定度）判断区域是否为“规整表格”；流程图/超密矩阵直接跳过，省掉分钟级空跑。
- **采纳闸门**：`config.yaml` 新增 `enhance_adopt`、`enhance_adopt_margin`、`adopt_sb_max`、`adopt_se_min`。采纳条件：`S_b < adopt_sb_max` 且 `(S_e - S_b) >= margin` 且 `S_e >= adopt_se_min`，三者全满足才 adopt，否则 compare。`table_enhancer._adopt_verdict()` 逐区域定；`markdown_generator` 采纳则替换表并去告警、加 `[!note]` 标注。
- **扩样本校准**：`_v3calib` 缓存扩到 13 个可用区域（6 成功出表 / 7 空 / 10 被适用性判据拦下），S_b 覆盖 0.43~0.997，逐区域人工比对内容。

**结论（关键）**：
1. **V3 对真正的表格页（如 121121）结构质量明显优于旧版 PP-Structure**（旧版整页误判 figure、切不出表）。这是接入它的价值所在。
2. **但 S_e 只测结构、不测内容。** 6 个成功样本中无一在内容上优于方案 B：121125 r02 S_e=0.886 但矩阵数字仍乱；121128 r04 gain +0.19 但表头错位、首列丢失；121134 r00 gain -0.11 反而更差。又宽又矮的窄带（高约 210~350px）V3 直接识别不出表、返回空。
3. **故阈值收紧到本样本零误采纳**：`margin 0.15→0.30`、`adopt_sb_max 0.62→0.55`、`adopt_se_min 0.80→0.90`（初版会误采纳内容错的 121125 r02）。闸门已就绪但当前语料不触发，待更强表格模型再放开。

**处置**：代码全部入库且默认关闭（`enhance_on_low_quality:false` + `enhance_adopt:false`）；33 张缓存回归逐字节等价 MATCH 33 / DIFF 0，`pytest tests/ -q` = 97 passed。提交 `ad655ca`（闸门实现）、`8301e99`（阈值校准）。与 7.18 “删代码回滚”不同，这里是“保留可插拔能力但默认不启用”。

## 6.7 方向 1A：并排合并宽表的语义拆分（调研 / 7.22）——未实施，归入待增强模型

**尝试方向**：针对 121134 / 121150 / 121157 这类“无天沟并排合并宽表”（方案 B 几何天沟法拆不开、质量分高于阈值不触发告警，属 7.9 遗留），尝试在几何/规则层加一条拆分修复，不依赖大模型。初始设想是“表头-数据重连”（把残缺表头表与相邻无表头数据表合并）。

**调研方法**：采集三张图的真实 block 结构（行分组、列锚点、最宽天沟比值）与最终渲染输出，不改代码。

**关键发现**：
1. **无贯穿全表的天沟。** 三张图的最宽列间隙/中位间隙比值仅 1.31~1.41，远低于 `_split_columns_by_gutter` 阈值 1.8——左右子表各自多列、列密集，中缝被正常列距淹没。
2. **121134 真因是三个子表被横向合并。** 右侧实为三块并排：`期限|敏感度值`（2列）+ `序号|权重|市值变化`（3列）+ `评级|行业|WEIGHT`（3列）。三块行高/行数各不相同（左块按期限分组、中块按序号 1-15、右块按行业），被按视觉行强行拼在一起时，同一 markdown 行的三段内容并不属于同一逻辑行。“表头-数据重连”对此不适用。
3. **121150 / 121157 的表格区域还混入了 PPT 工具栏噪声 + 标题 + 公式**，且同样是左右双表结构。

**结论**：正确拆分依据是语义（三块各是什么含义），不是几何。任何足以拆开它们的规则都极可能误伤当前正常的宽表（如 121134 左侧实体表现已正确，一动就可能弄坏），与“每张变好或不变差”硬约束冲突。这与 7.9 / 方案 A / 方向 3 撞到的是同一堵墙。

**处置**：几何层不再为这类图打补丁；121134/121150/121157 正式归入“待更强模型（方向二，Mac）+ 已就绪的采纳闸门”处理。本轮未改任何代码。

## 7. 跨方向的共性教训

1. **彩色幻灯片语料上，PP-Structure/gridboost 无法稳定优于几何法（方案 B）。** 三次尝试（7.15/7.16/7.17）结论一致，故都退为 review-only。
2. **VLM 重建的真正难点是“采纳判据”，不是模型能不能看。** 字数覆盖率（7.18）会放行被篡改的表格；逐 token 不漏不造（7.20）才是可靠信号。
3. **本机 CPU + 3B 模型是硬上限。** 单区域分钟级耗时，且 3B 在高分裁剪下仍漏内容（hit=0.16）。要真正采纳需更强模型（方向 1）。
4. **并排合并宽表（121134/121150/121157）是结构性难点。** 列间无几何天沟、质量分高于阈值不触发告警，几何法无法拆，唯一区分依据是语义。
5. **纯附加、默认关闭、全量回归逐字节等价，是每次尝试的安全底线。** 正因如此，上述所有失败方向都未对现有识别质量造成任何负面影响。

6. **结构分（S_e）不能当采纳依据，采纳闸门宁可不触发也不误采纳。** PP-StructureV3（7.21）再次验证 7.18 教训：结构规整不代表内容对。阈值校准的正确目标不是“找一个能采纳的阈值”，而是“在本样本内绝不误采纳”。
